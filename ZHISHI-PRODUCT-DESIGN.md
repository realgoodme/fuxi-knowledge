## 一、产品定义

| 维度 | 内容 |
|------|------|
| **定位** | 一个专给自己用的 AI 知识库 |
| **一句话** | 给写公众号的人用的 AI 助手——素材丢进去，AI 读完写摘要、建索引、往回织关联、告诉我今天能写哪篇 |
| **用户** | 我自己（一个人用） |
| **解决的问题** | 素材散落各处、选题时找不到、想发散没人帮、AI 完全没参与写作流程 |

---

## 二、MVP 定义

### 一页纸 MVP

| 格子 | 内容 |
|------|------|
| **用户** | 我自己 |
| **问题** | 素材散落、选题找不到、想发散没人帮、AI 没参与 |
| **输入** | 我手动丢进去的链接、PDF、想法 |
| **处理** | AI 读完 → 写带立场的摘要 → 更新索引 → 往回织关联条目 |
| **输出** | 一个能问、能查、能告诉我「今天可以写哪篇」的本地知识库 |
| **不做什么** | 不做数据库、不做配图、不做协作、不做多端、不做音视频识别 |

### MVP 验证标准（用 14 天自测）

- ✅ 每天主动想去用它
- ✅ 用了一周产生「再加 XX 功能就完美了」的冲动
- ✅ 两周后沉淀了至少 20 条能用的素材 / 摘要

## 三、架构设计（进存取）

### 数据流

```
输入                   存储（分层）                    输出
────────────────────────────────────────────────────────────
                      ┌─────────────┐
链接 / URL            │  原料层      │
  │                   │  raw/ 原文   │
  ├──→ 抓取            │  只读不改    │
  │                   └──────┬──────┘
PDF / 文件                   │
  │                  ┌───────▼───────┐
  ├──→ 读取           │  笔记层        │
  │                   │  notes/       │
想法 / 灵感           │  带立场摘要    │
  │                   └───────┬───────┘
  ├──→ 记录                    │
                             │  ← ← ← 往回织
                     ┌───────▼────────┐
                     │  编译层          │
                     │  wiki/          │
                     │  • 主题页        │  ← ← 问答流
                     │  • 实体页        │  ← ← 选题流
                     └────────────────┘  ← ← 周报流
```

### 进（输入）

| 维度 | 内容 |
|------|------|
| **来源** | 公众号文章链接、GitHub 链接、X 链接、PDF 文件、自己冒的想法 |
| **触发** | 手动丢给 AI（暂不做自动抓取） |
| **格式** | URL / PDF / 自然语言 |

### 存（四层存储）

| 层级 | 目录 | 存什么 | 规则 |
|------|------|--------|------|
| ❶ 原料层 | `raw/` | 抓回来的原文、PDF | **只读不改**，事实来源 |
| ❷ 笔记层 | `notes/` | AI 写的带立场摘要，每篇一条 | 写入，不覆盖原料 |
| ❸ 编译层 | `wiki/` | 主题页（多篇笔记合并结论）+ 实体页（同一事物所有提及汇总） | AI 写，往回织时重写 |
| ❹ 索引层 | `index.md` | 全库目录 + 每页一句话摘要 | 脚本自动更新 |

**核心机制：往回织**
每加一条新素材，AI 回头扫描相关主题页、实体页，更新结论、标注矛盾。不是堆叠，是往里织。

### 取（三条输出流）

| 流 | 谁取 | 何时取 | 取什么 |
|----|------|--------|--------|
| ❶ **问答流** | 我 | 随时问 | AI 查索引 → 读主题页 → 综合回答 |
| ❷ **选题流** | 我 | 想写文章时 | 一份选题清单（基于主题页推荐可写方向） |
| ❸ **周报流** | 我/AI | 每周 | 本周入库汇总 + 哪些主题需要编译 |

## 四、技术栈

### 整体原则

> **有什么 AI 用什么，不锁定。本地优先，简单优先。**

当前可用的 AI：DeepSeek、智谱 GLM、Claude、GPT 等，根据任务灵活调度。

### 抓取层

| 渠道 | 做法 | 原因 |
|------|------|------|
| 普通网页 | `curl` + AI 提取 | 零依赖 |
| 微信公众号 | 现有抓取方案 | 反爬严，用现成的 |
| GitHub | `gh CLI` | 官方命令行 |
| YouTube | `yt-dlp` | 社区成熟工具 |
| X / Twitter | 社区工具 | 反爬最狠，不造轮子 |

### 存储层

| 项目 | 方案 | 原因 |
|------|------|------|
| 主数据 | **纯 Markdown 文件** | 人能读、git 能 diff、AI 能直接喂 |
| 目录分层 | 按四层组织（raw / notes / wiki / index）| 按生熟程度，不按时间 |
| 数据库 | **不用** | 个人知识库，文件够 |
| 向量库 | **不用** | 用标签重叠 + 关联推荐替代 |

### 索引 + 编排层

| 项目 | 方案 |
|------|------|
| 自动索引 | Python 脚本，扫 frontmatter 生成 index.md |
| 关联推荐 | 打分脚本（标签重叠 + 标题词重叠 → 加权打分 → 双向写入 related 字段） |
| 检索 | 关键词搜索 + AI 长上下文 |
| AI 模型 | 不锁定，有什么用什么，按任务灵活调度 |
| 运行环境 | 本地（零服务器、零部署、零运维） |

## 五、不做清单

- ❌ 不做数据库（文件够）
- ❌ 不做向量库 / embedding / RAG（标签重叠法够用）
- ❌ 不做配图 / 封面生成
- ❌ 不做协作 / 多端同步
- ❌ 不做音视频识别
- ❌ 不做 web UI（命令行 + AI 会话就够了）
- ❌ 不做自动抓取（先手动跑通再考虑）

## 六、开工前铁律

1. **先画「取」再画「进」** — 倒推设计，先想清楚输出什么，再决定存什么、怎么进
2. **分层 = 给数据上保险** — 原料层只读，AI 错了能回退
3. **功能做 MVP，架构想两条** — 能长大（分层留扩展口）+ 不会塌（数据不丢、能回退）
4. **机械归脚本，语义归 AI** — 扫文件/统计/生成索引用脚本；判断关联/打标签/写摘要用 AI
5. **有什么 AI 用什么** — 不锁定模型，灵活调度
